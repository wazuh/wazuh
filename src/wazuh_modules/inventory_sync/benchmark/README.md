# Inventory Sync Benchmark Tools

Tools for measuring the memory, CPU, and throughput impact of the
`inventory_sync` module under load. Designed for **before/after** comparison
when applying production changes (queue limits, instrumentation,
threading adjustments, etc.).

> **Current scope: measurement only.**
> These tools do not modify the module's behaviour. They do not add
> logs, counters, or limits in production. They only observe the
> `wazuh-modulesd` process, the traffic generated by synthetic agents,
> and the patterns the module already emits to `wazuh-manager.log`.

## Layout

```
inventory_sync/
├── shared/                      # Single source for QA and benchmark
│   ├── flatbuffers_manager.py
│   ├── generate_flatbuffers.py
│   ├── agent_controller.py
│   └── requirements.txt
├── qa/                          # Integration tests (untouched)
│   └── ...
└── benchmark/                   # This.
    ├── run_benchmark.sh         # Single orchestrator
    ├── benchmark_sender.py      # Multi-agent load generator
    ├── monitor.py               # Process monitor (RSS/VMS/CPU/FDs/...)
    ├── log_parser.py            # Tail of wazuh-manager.log + counters
    ├── result_summary.py        # Merges CSVs and emits summary.json + PASS/FAIL
    ├── graphics_generator.py    # Comparison charts
    ├── generate_payloads.py     # Synthetic payloads of given size
    ├── indexer_control.sh       # Wrapper for service wazuh-indexer
    ├── cleanup_agents.sh        # Deletes bench-* agents via API
    ├── debug_single_session.py  # Single-session debug
    ├── requirements.txt         # Benchmark-specific deps
    ├── scenarios/               # Declarative scenarios (JSON)
    └── sample_payloads/
```

## Setup

```bash
cd inventory_sync/benchmark
pip install -r requirements.txt
pip install -r ../shared/requirements.txt

# Generate FlatBuffers classes (the sender also invokes this if missing)
python3 ../shared/generate_flatbuffers.py
```

`shared/` contains `flatbuffers_manager`, `generate_flatbuffers` and
`agent_controller`. Both `qa/` and `benchmark/` import from there,
adding `inventory_sync/shared` to `sys.path` in each entry point.

## Quick start

### Scenario mode (recommended)

```bash
./run_benchmark.sh --scenario scenarios/baseline.json --manager 127.0.0.1
./run_benchmark.sh --scenario scenarios/burst.json    --manager 127.0.0.1
```

### Manual mode (no scenario)

```bash
./run_benchmark.sh \
    -a 50 -d 200 -t 60 \
    --label baseline \
    --manager 127.0.0.1
```

### Comparing runs

```bash
./run_benchmark.sh --compare results_baseline results_burst
```

## What each run produces

Under `results_<label>/`:

| File                     | Source                | Contents |
|---|---|---|
| `bench.csv`              | `benchmark_sender.py` | Per-second counters (messages, disaggregated ACKs, ReqRet, missing ranges, drops) |
| `sender_summary.json`    | `benchmark_sender.py` | Totals + percentiled latencies (Start→StartAck, End→EndAck, full session) |
| `monitor.csv`            | `monitor.py`          | Per-second process stats: RSS, VMS, CPU%, mem%, threads, FDs, IO, uptime |
| `logs.csv`               | `log_parser.py`       | Per-second counts of existing patterns in `wazuh-manager.log` |
| `summary.json`           | `result_summary.py`   | Final executive summary + PASS/FAIL checks against `scenario.expectations` |
| `params.json`            | `run_benchmark.sh`    | Exact run configuration |
| `system_info.txt`        | `run_benchmark.sh`    | Host, kernel, CPU, RAM |
| `charts/`                | `graphics_generator`  | PNG/SVG/PDF |

## What is measured and what is not

### Yes (without production instrumentation)

- **Process**: `pid`, `uptime_sec`, `cpu_pct`, `mem_pct`, `rss_mb`, `vms_mb`,
  `threads`, `fds`, `read_bytes`, `write_bytes`.
- **Disk**: recursive size in MB of each tracked directory.
  By default `run_benchmark.sh` tracks:
  - `/var/wazuh-manager/queue/inventory_sync` — RocksDB of the Inventory
    Sync module (grows with in-flight sessions, shrinks on close / timeout).
  - `/var/wazuh-manager/queue/engine-output` — engine output queue.
  - `/var/wazuh-manager/queue/vd` — Vulnerability Detector database.
  Each path adds a `dir_<basename>_mb` column to `monitor.csv` and a
  `disk.<basename>` block to `summary.json` with min/max/final/growth.
  Configurable with `--disk-path PATH` (repeatable) in `run_benchmark.sh` or
  `monitor.py`.
- **Sender**: `messages_sent`, `sessions_started/completed/failed`,
  `start_ack_ok/offline/error`, `end_ack_ok/offline/error/processing`,
  `reqret`, `missing_ranges_total`, `messages_dropped`.
- **Latencies** per session (in the sender): p50/p90/p95/p99/max for
  Start→StartAck, End→EndAck, and the full session.
- **Logs** from the manager (existing patterns): `session_limit_reached`,
  `zombie_cleaned`, `session_timeout`, `vdsync_timeout`, `parse_error`,
  `module_check_failed`, `inventory_sync_error`, `indexer_error`.

#### Disk expectations in scenarios

Each scenario can declare per-directory limits:

```json
"expectations": {
  "disk_growth_mb_max": {
    "inventory_sync": 100,
    "engine_output":  20,
    "vd":             50
  }
}
```

`result_summary.py` evaluates them at the end of the run and emits them as
PASS/FAIL checks in `summary.json`. Useful for detecting leaks in the module's
RocksDB (sessions that don't close their prefix) or uncontrolled growth
of downstream queues.

### No (would require new instrumentation)

- Current or maximum size of `m_workersQueue`, `m_indexerQueue`,
  `m_responseDispatcher`.
- Drop counters for full queues.
- "queue full" or "high watermark" warnings.

These fields would always be 0 today. When a PR exposing them lands
(stats socket, high watermark log, etc.), `log_parser.py` and
`monitor.py` can be extended without touching the rest.

### How to infer overflow

Without instrumentation, the indirect clues are:

| Symptom | Probable cause |
|---|---|
| `messages_sent` >> `end_ack_ok × messages_per_session` | Drops before the worker |
| sustained `reqret > 0` | `DataValue` drops (GapSet detects gaps at `End`) |
| linearly rising `rss_mb` with stable load | Some internal queue growing unbounded |
| `latency_p99` spikes with saturated CPU | Bottleneck in `m_indexerQueue` (single-thread) |
| `start_ack_offline > 0` + log `Session limit reached` | `m_maxSessions` saturated |
| `end_ack_error > 0` with `indexer_error` in logs | Indexer down or slow |

## Available scenarios

Each JSON in `scenarios/` defines `load` (parameters) and `expectations`
(thresholds for PASS/FAIL). Adjust the thresholds according to local SLOs.

| File                                 | Load / purpose |
|---|---|
| `scenarios/baseline.json`            | Normal load with `package` payload. No drops expected. |
| `scenarios/baseline_fim.json`        | Identical to baseline but with `payload_kind: fim_file` (→ `wazuh-states-fim-files`). |
| `scenarios/intra_session_race.json`  | Triggers the intra-session race between `handleEnd` and `handleData` in parallel workers. ReqRet > 0 expected. |
| `scenarios/burst.json`               | 200 agents, 500 msgs, no pause. |
| `scenarios/indexer_down.json`        | Indexer stop at 30s. |
| `scenarios/missing_seq.json`         | `drop_every` to force ReqRet (skips seq=N-1, 2N-1, ...). |
| `scenarios/no_end.json`              | No `End` to validate `session_timeout` and RocksDB cleanup. |
| `scenarios/databatch.json`           | `MessageType_DataBatch` path (per-element re-serialization). |
| `scenarios/large_payload.json`       | 64 KB payloads to validate the implicit `m_bulkDataSize=10MB` limit. |
| `scenarios/soak.json`                | 1h to detect memory leaks. |
| `scenarios/mega_burst.json`          | **OOM**: 1000 agents saturate the `m_workersQueue` producer (unlimited). |
| `scenarios/indexer_down_burst.json`  | **OOM**: 500 agents + downed indexer → simultaneous pressure on `m_workersQueue` and `m_indexerQueue`. |
| `scenarios/heavy_payload_burst.json` | **OOM**: 100 agents × 1 MB/DataValue → per-element weight. |
| `scenarios/sustained_high_rate.json` | **OOM**: 300 agents × 1h sustained burst → chronic backpressure. |

Each scenario can declare `load.payload_kind` (`package`, `system`,
`hotfix`, `fim_file`, `sca_check`). The CLI always wins: if you pass
`--payload-kind fim_file` the scenario respects it.

### Running baseline with FIM payload (two ways)

```bash
# Option 1: dedicated scenario (more declarative)
./run_benchmark.sh --scenario scenarios/baseline_fim.json

# Option 2: CLI override on the normal baseline
./run_benchmark.sh --scenario scenarios/baseline.json \
                   --payload-kind fim_file \
                   --label baseline_fim_override
```

Either one sets module `fim`, index `wazuh-states-fim-files`, and
uses `sample_payloads/fim_file.json`. Verifiable post-run:

```bash
curl -sk -u admin:admin 'https://localhost:9200/wazuh-states-fim-files*/_count?pretty'
```

### Additional sender flags (inherited from scenario or via CLI)

Each flag is read from the `load.*` field of the scenario JSON or can be
passed manually to `run_benchmark.sh`. The CLI wins over the scenario.

| Flag (`run_benchmark.sh` / `benchmark_sender.py`) | Field in `load` | What it does |
|---|---|---|
| `--payload-size N`     | `payload_size_bytes` | Inflates each DataValue by extending an existing free-text field (chosen per `--payload-kind` from `PAD_FIELD_BY_KIND`; e.g. `file.path` for `fim_file`) until the JSON reaches ≥ N bytes. Adding a new top-level field would be rejected by the dynamic:strict mappings (`HTTP 400 mapping set to strict, dynamic introduction of [_pad] within [_doc] is not allowed`). **Hard ceiling ~50 KB**: remoted's uncompress buffer is `OS_MAXSTR=65536` (`shared/include/defs.h:51`); larger plaintext triggers `(2202) Error uncompressing string` and the connection is closed before reaching `m_workersQueue`. To stress queue weight beyond that, scale `agents`/`data_size` rather than per-message size. |
| `--pad-field PATH`     | `pad_field`          | Dotted path of the payload field to inflate. Overrides the per-kind default. Must point to an existing string field in the mapping. |
| `--drop-every N`       | `drop_every`         | Skips DataValues where `(seq+1) % N == 0`. Forces ReqRet/missing_ranges. |
| `--no-end`             | `no_end: true`       | Does not send End or wait for EndAck. The session stays open and `session_timeout` reclaims it. |
| `--use-databatch`      | `use_databatch: true`| Sends DataValues as `Message{DataBatch{DataValue[]}}` in batches. |
| `--batch-size N`       | `batch_size`         | DataValues per batch when `--use-databatch` is active (default: 64). |

## Agent cleanup

`run_benchmark.sh` performs two cleanups with different behaviour:

- **Pre-run (always)**: deletes `bench-*` agents left over from
  previous runs. Prevents accumulation.
- **Post-run (opt-in with `--cleanup-after`)**: by **default does NOT** delete
  agents at the end of the run. If it did, [InventorySyncFacade::deleteAgent](../src/inventorySyncFacade.hpp)
  would trigger `deleteByQuery("wazuh-states-*", agentId)` and the indexed
  documents would disappear — the IT Hygiene dashboard would look empty.

If docs are visible in `_cat/indices` with `docs.count: 0` but
`docs.deleted: N`, that is a symptom of an aggressive post-run cleanup. Fix:
don't pass `--cleanup-after` (which is the default).

```bash
# Inspection post-run (default):
./run_benchmark.sh --scenario scenarios/baseline.json
# Afterwards: bench-* agents remain registered.
curl -sk -u admin:admin 'https://localhost:9200/wazuh-states-inventory-packages*/_count?pretty'
# And they appear in IT Hygiene.

# Explicit cleanup for CI:
./run_benchmark.sh --scenario scenarios/baseline.json --cleanup-after
```

## Before/after comparison

```bash
# Before the change
./run_benchmark.sh --scenario scenarios/burst.json --label burst_before

# Apply the change (production PR, deploy, config adjustment...)

# After
./run_benchmark.sh --scenario scenarios/burst.json --label burst_after

# Visual diff
./run_benchmark.sh --compare results_burst_before results_burst_after
```

`summary.json` from each run has the same fields, so they lend themselves
to a quick `jq`:

```bash
jq '.result, .process.rss_mb_max, .latency_ms.session_full.p99' \
    results_burst_before/summary.json results_burst_after/summary.json
```

## Re-process an existing run

The CSVs are the source of truth. To regenerate `summary.json` for an
old run:

```bash
python3 result_summary.py \
    --bench       results_burst_before/bench.csv \
    --monitor     results_burst_before/monitor.csv \
    --logs        results_burst_before/logs.csv \
    --sender-json results_burst_before/sender_summary.json \
    --scenario    scenarios/burst.json \
    --params      results_burst_before/params.json \
    --out         results_burst_before/summary.json
```

## Notes

- The sender uses the real path `auth(1515) + remoted(1514)` with AES + zlib.
  Saturating the local manager consumes host RAM/CPU; have at least 4 GB
  free before running `burst.json`.
- `log_parser.py` starts at EOF of `/var/wazuh-manager/logs/wazuh-manager.log`;
  to post-process an existing log, pass `--from-start`. Change the
  path with `--log` or with `--manager-log` from `run_benchmark.sh`.
- `results_*/` is meant to live outside git. The already-committed
  `results_no-limits/` is a historical reference of the pre-fix state.

### Payload ↔ index alignment (FIM / SCA / syscollector)

The payload must match **exactly** the target index mapping.
The `wazuh-states-*` indices have `dynamic: strict`, so any
undeclared field triggers HTTP **400** from the indexer:

```
mapping set to strict, dynamic introduction of [<field>] within [<parent>] is not allowed
```

This shows up in `/var/wazuh-manager/logs/wazuh-manager.log` and is counted
by `log_parser.py` as `indexer_error`. If you see it, **the benchmark is NOT
measuring module capacity** — it is measuring indexing errors.

To avoid this, the sender exposes `--payload-kind` which selects a consistent
payload + module + index:

| `--payload-kind` | Payload sample                      | `--module`     | `--index` target                   | Root fields |
|---|---|---|---|---|
| `package`        | `syscollector_package.json`         | `syscollector` | `wazuh-states-inventory-packages`  | `package.*` (architecture, category, description, installed, multiarch, name, path, priority, size, source, type, vendor, version) |
| `system`         | `syscollector_system.json`          | `syscollector` | `wazuh-states-inventory-system`    | `host.*` (architecture, hostname, os.*) |
| `hotfix`         | `syscollector_hotfix.json`          | `syscollector` | `wazuh-states-inventory-hotfixes`  | `package.hotfix.name` |
| `fim_file`       | `fim_file.json`                     | `fim`          | `wazuh-states-fim-files`           | `file.*` (path, hash.{md5,sha1,sha256}, attributes, device, gid, group, inode, mtime, owner, permissions, size, uid) |
| `sca_check`      | `sca_check.json`                    | `sca`          | `wazuh-states-sca`                 | `check.*`, `policy.*` |

All sample payloads also include `checksum.hash.sha1`,
`state.document_version` and `state.modified_at` (required by all
mappings). The `wazuh.agent.*` and `wazuh.cluster.*` fields are injected
by the manager when building the bulk; the agent payload **must not** include them.

Examples:

```bash
# Default: --payload-kind package
./run_benchmark.sh -a 10 -t 60

# FIM
python3 benchmark_sender.py --payload-kind fim_file -a 10 -d 100 -t 60 \
    --manager 127.0.0.1 -o bench_fim.csv

# SCA
python3 benchmark_sender.py --payload-kind sca_check -a 10 -d 100 -t 60 \
    --manager 127.0.0.1 -o bench_sca.csv

# Custom payload (overrides everything if you pass --module/--index)
python3 benchmark_sender.py \
    --payload my_payload.json \
    --module syscollector \
    --index  wazuh-states-inventory-processes \
    -a 10 -d 50 -t 30
```

For arbitrary-sized payloads:

```bash
python3 generate_payloads.py --kind fim_file --size 8192 -o big_fim.json
python3 benchmark_sender.py --payload big_fim.json --module fim \
    --index wazuh-states-fim-files -a 10 -t 60
```

`generate_payloads.py --kind` acepta los mismos 5 valores que
`--payload-kind` y respeta los mappings reales.
